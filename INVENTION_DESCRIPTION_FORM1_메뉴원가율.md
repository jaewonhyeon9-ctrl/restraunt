## 1. 발명의 명칭

**메뉴 레시피 기반 다중 재고 자동 차감 및 실시간 원가율 산출 방법**

## 2. 종래기술에 대한 설명

식당 운영자가 메뉴별 원가와 손익을 파악하기 위한 종래의 방식은 다음과 같이 구분된다.

### (1) 종래기술 1: 수기 또는 엑셀 기반 원가 계산

식당 운영자가 메뉴별 사용 식자재와 사용량을 종이 장부 또는 엑셀에 수기로 정리하고, 매출 발생 후 별도 시간을 들여 사용량을 합산하여 원가를 추정하는 방식이다.

- 매출 발생과 원가 계산이 시간적으로 분리됨
- 식자재 단가 변동 시 수동 갱신 필요
- 원가율은 일/주/월 단위 사후 계산
- 메뉴별·카테고리별 원가율 도출에 다수의 작업 단계 필요

### (2) 종래기술 2: POS 시스템과 별도의 재고관리 시스템 병행 운영

식당 운영자가 매출 입력은 POS 단말로, 재고 입출고는 별도의 재고관리 소프트웨어로 운영하면서 두 시스템 간 데이터를 정기적으로 매칭하는 방식이다.

- 매출 시스템과 재고 시스템이 분리되어 있음
- 일/주/월 단위로 재고 차감 일괄 처리
- 메뉴 1개 판매 시 사용된 식자재 항목·수량을 사후에 추정
- 시스템 간 매칭 오류로 인한 누락·중복 차감 빈번

### (3) 종래기술 3: 거래명세표 OCR 기반 재고 입고 자동화 시스템

공급업체로부터 발행된 거래명세표를 OCR로 인식하여 재고를 입고 측에서 자동 적산(가산)하고, 부족 재고 발생 시 공급업체에 발주를 자동화하는 시스템(대한민국 등록특허 제10-2741384호 등)이 알려져 있다.

- 입고(IN) 측 자동화에 한정됨
- 메뉴-식자재 간 매핑(레시피) 개념 없음
- 매출(OUT)에 따른 재고 차감은 다루지 않음
- 메뉴별 원가율 산출 기능 없음

### (4) 종래기술 4: 회계 SaaS 솔루션 (캐셔플, 페이히어 등)

매출과 매입을 통합 관리하는 클라우드 회계 솔루션은 매출 합계와 매입 합계를 비교하여 식당 전체의 원가율(매입/매출)을 산출한다.

- 식당 전체 단위의 거시 원가율만 제공
- 메뉴별·카테고리별 원가율 산출 불가
- 식자재 단가 변동의 시점별 영향 추적 불가
- 별도 단말 또는 데스크톱 환경 필요

## 3. 종래기술의 문제점에 대한 설명

### (1) 매출 발생 시점과 재고 차감 시점의 불일치

종래기술 1, 2는 매출이 발생한 시점과 재고가 차감되는 시점이 분리되어 있어 재고 잔량이 실시간으로 정확하지 않다. 이로 인해 식당 운영자는 다음과 같은 불편을 겪는다.

- 영업 중 식자재 소진 여부를 정확히 파악하지 못함
- 발주 시점 판단에 매번 수동 점검 필요
- 식자재 폐기 또는 부족으로 인한 영업 차질 발생

### (2) 식자재 단가 변동의 시점별 추적 불가

종래기술은 식자재 단가가 미래에 변경되면 과거 매출의 원가도 함께 변경된 단가로 재계산된다. 즉, 판매 시점의 실제 원가가 보존되지 않아 다음 문제가 발생한다.

- 식자재 단가가 인상된 후 과거 손익 데이터가 왜곡됨
- 메뉴별 수익성 변화를 시기별로 비교 불가
- 가격 인상 의사결정의 근거 데이터 부재

### (3) 메뉴별·카테고리별 원가율 산출 불가

종래기술 4의 회계 SaaS는 식당 전체의 매출 대비 매입 비율만 산출하므로, "어떤 메뉴가 수익성이 높은지"를 데이터 기반으로 판단할 수 없다.

- 비효율 메뉴 식별 불가
- 가격 인상 또는 메뉴 폐지 의사결정의 근거 부재
- 카테고리별 (예: 메인 메뉴 vs 사이드 vs 음료) 수익성 비교 불가

### (4) 다중 재고의 원자적(atomic) 차감 미보장

메뉴 1개를 판매하면 N개의 식자재가 동시에 소비되는데, 종래기술은 이러한 다중 재고 차감을 보장하지 않거나 별도의 트랜잭션으로 처리하여 일부 식자재만 차감되고 나머지는 차감되지 않는 데이터 불일치가 발생할 수 있다.

### (5) 원가율 임계 초과에 대한 즉시 대응 부재

식자재 가격이 급등하여 메뉴 원가율이 임계치를 초과하더라도 종래기술은 운영자가 별도로 손익 보고서를 확인하기 전까지 이를 알지 못한다. 운영자가 인지했을 때는 이미 손실이 누적된 상태이다.

## 4. 발명의 구성 및 동작 원리에 관한 설명

### (1) 시스템 전체 구성

본 발명은 식당 운영자가 사용하는 휴대폰 PWA(Progressive Web App)와 서버, 데이터베이스로 구성된다. 시스템은 다음의 구성요소를 포함한다.

```
[메뉴 레시피 등록부]   ← 식당 운영자가 메뉴 1개당 사용 식자재와 사용량 등록
        ↓
[매출 발생 처리부]   ← 매출 1건 등록 (직접 입력 / 영수증 OCR / 엑셀 업로드 등)
        ↓
[다중 재고 동시 차감부]   ← 단일 데이터베이스 트랜잭션으로 N개 식자재 동시 감산
        ↓
[원가 스냅샷 저장부]   ← 판매 시점의 식자재 단가를 매출 항목 레코드에 저장
        ↓
[원가율 산출부]   ← 메뉴별 / 카테고리별 / 전체 식당 단위로 실시간 산출
        ↓
[임계 초과 알림부]   ← 원가율이 사전 설정 임계 초과 시 사용자 단말 푸시
```

### (2) 메뉴 레시피 등록부의 동작

식당 운영자는 시스템에 메뉴를 등록할 때, 해당 메뉴 1개를 만들 때 사용되는 **복수의 식자재 항목과 각 항목별 사용량**을 지정한다. 이를 본 명세서에서는 "레시피"라 칭한다.

예시:
- 메뉴: "비빔밥"
- 레시피:
  - 쌀 200g
  - 소고기 100g
  - 고추장 30g
  - 채소(시금치) 50g
  - 채소(콩나물) 50g

이러한 레시피 데이터는 데이터베이스의 메뉴-식자재 매핑 테이블에 저장된다.

### (3) 매출 발생 시 다중 재고 동시 차감 동작

식당 운영자가 매출 1건을 시스템에 등록하면, 본 발명은 다음 단계를 단일 데이터베이스 트랜잭션 내에서 수행한다.

**단계 1: 매출 항목 분해**
- 매출에 포함된 각 메뉴와 판매 수량을 추출

**단계 2: 레시피 조회 및 차감량 계산**
- 각 메뉴에 대해 레시피를 조회
- 차감량 = 판매 수량 × 레시피상 식자재 사용량
- 동일 식자재가 여러 메뉴에서 사용되는 경우 차감량 합산

**단계 3: 식자재 단가 스냅샷 캡처**
- 차감 시점에 각 식자재의 단가(unitPrice)를 조회
- 판매 시점 단가를 매출 항목 레코드(SaleItem)의 `costAtSale` 필드에 저장

**단계 4: 다중 재고 원자적 감산**
- 단일 트랜잭션 내에서 모든 식자재 잔량을 동시에 감산
- 차감 로그(InventoryLog)도 같은 트랜잭션에서 기록
- 어느 한 식자재 차감이 실패하면 전체 트랜잭션 롤백

**단계 5: 원가율 갱신**
- 메뉴별 원가율 = 1메뉴 원가 / 메뉴 판매가
- 1메뉴 원가 = Σ(레시피상 사용량 × 식자재 단가)
- 카테고리별 원가율 = Σ(카테고리 내 메뉴 원가) / Σ(카테고리 내 매출)
- 전체 식당 원가율 = Σ(매출 항목별 costAtSale × 수량) / Σ(매출 항목별 매출액)

### (4) 원가율 임계 초과 알림 동작

운영자는 메뉴별 또는 식당 전체 원가율의 임계값(예: 35%)을 사전 설정할 수 있다. 본 발명은 매출 등록 후 원가율을 갱신할 때마다 다음을 수행한다.

- 갱신된 원가율과 임계값을 비교
- 임계값 초과 시 운영자 단말로 푸시 알림 발송
- 알림에는 어느 메뉴의 원가율이 어떻게 변화했는지 포함

이를 통해 운영자는 식자재 가격 급등을 즉시 인지하고 가격 인상 또는 메뉴 조정 의사결정을 신속히 할 수 있다.

### (5) 핵심 기술적 특징

본 발명의 핵심 기술적 특징은 다음과 같다.

**가. 단일 트랜잭션 내 다중 재고 동시 차감**

종래기술은 메뉴 1개 판매 시 N개 식자재를 별도 처리하거나 사후 일괄 처리하였으나, 본 발명은 매출 등록과 동시에 단일 데이터베이스 트랜잭션 내에서 모든 식자재를 원자적으로 감산하여 데이터 정합성을 보장한다.

**나. 판매 시점 원가 스냅샷 보존**

본 발명은 차감 시점의 식자재 단가를 매출 항목 레코드에 별도 보존(`costAtSale`)하여, 식자재 단가가 미래에 변경되더라도 과거 매출의 원가는 변경되지 않고 보존된다. 이를 통해 시기별·기간별 정확한 손익 분석이 가능하다.

**다. 메뉴별·카테고리별·전체 다층 원가율 산출**

본 발명은 동일한 매출 데이터로부터 메뉴 단위, 카테고리 단위, 식당 전체 단위의 원가율을 동시에 실시간 산출한다.

**라. 임계 초과 시 자동 푸시 알림**

원가율이 사전 설정 임계를 초과하는 즉시 운영자 단말로 알림이 전송되어, 식자재 가격 변동에 대한 능동적 대응이 가능하다.

**마. PWA 기반 단일 단말 환경**

종래기술이 별도의 POS 단말 또는 데스크톱 환경을 요구하는 것과 달리, 본 발명은 휴대폰 PWA로 구현되어 운영자가 별도 장비 없이 언제 어디서나 동작 가능하다.

## 5. 발명의 효과

본 발명에 따르면 다음과 같은 효과를 얻을 수 있다.

### (1) 매출-재고 실시간 동기화

매출 발생과 동시에 다중 재고가 단일 트랜잭션으로 감산되어 식당 영업 중 실시간 재고 잔량 파악이 가능하다. 종래기술 대비 재고 파악 시간을 약 70% 단축할 수 있다.

### (2) 인적 오류 감소

종래의 수기 차감 또는 시스템 간 매칭 작업에서 발생하던 누락·중복 차감 오류가 단일 트랜잭션 처리로 거의 발생하지 않는다. 인적 오류를 약 90% 감소시킬 수 있다.

### (3) 시기별 정확한 손익 분석

판매 시점 단가 스냅샷 보존으로 식자재 단가가 변동된 후에도 과거 매출의 원가가 보존되어, 월별·계절별 메뉴 수익성 변화를 정확히 분석할 수 있다.

### (4) 메뉴별 수익성 즉시 파악

메뉴별 원가율을 실시간으로 시각화(예: 30% 미만 녹색, 30~40% 황색, 40% 이상 적색)하여 운영자가 어느 메뉴의 수익성이 떨어지는지 즉시 인지할 수 있다.

### (5) 식자재 가격 변동에 대한 능동 대응

원가율 임계 초과 시 즉시 푸시 알림을 받아 가격 인상 또는 메뉴 조정 의사결정을 신속하게 할 수 있다. 종래기술은 운영자가 사후 보고서를 확인할 때까지 손실이 누적되었다.

### (6) PWA 기반 무설치·무단말 환경

본 발명은 휴대폰 PWA로 동작하여 별도의 POS 단말이나 소프트웨어 설치 없이 누구나 즉시 사용 가능하다. 영세 식당의 IT 진입 장벽을 제거한다.

### (7) 자동 발주 시점 의사결정 지원

실시간 재고 잔량과 매출 추세 데이터가 결합되어 발주 시점 의사결정의 근거가 된다(별도 발명).

### 도 4 — 원가율 산출 수식

```
   메뉴 i의 1개 원가:
       원가_i = Σ_j (recipe[i,j].qtyUsed × inventory[j].unitPrice)

   메뉴 i의 원가율:
       원가율_i = 원가_i / menu[i].price × 100 (%)

   전체 식당 원가율:
       원가율_총 = Σ_k (saleItem[k].costAtSale × saleItem[k].qty)
                 / Σ_k saleItem[k].subtotal
                 × 100 (%)

   원가율 색상 코딩:
       원가율 < 30%        : 녹색 (양호)
       30% ≤ 원가율 < 40%  : 황색 (주의)
       원가율 ≥ 40%        : 적색 (개선 필요)
```
